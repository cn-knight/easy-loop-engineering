## 一、Loop Engineering（循环工程）概念

### 1. 是什么

继 **Prompt Engineering（提示词工程）** 和 **Harness Engineering（脚手架工程）** 之后，在 AI 编程 Agent（Claude Code、Codex 等）领域新出现的前沿概念。

- **核心思想**：开发者不再亲自给编程 Agent 写提示词，而是设计 **Loop（循环/闭环工作流）**，让 Loop 替开发者去写提示词、分发任务、检查结果。
- OpenClaw 创始人 Peter Steinberger："你不该再亲自给编程 Agent 写提示词了。你应该设计 Loop，让 Loop 替你去写。"
- Claude Code 之父 Boris Cherny："我已经不给 Claude 写提示词了，我有 Loop 在跑，它们自己决定该干什么，我的工作是写 Loop。"

### 2. 与传统方式的对比

| 维度 | 传统方式（Prompt Engineering） | Loop Engineering |
|------|------------------------------|------------------|
| 流程形态 | 线性：写提示词 → 塞上下文 → 回车 → 读结果 → 再敲下一句 | 自行运转的闭环 |
| 人与 Agent 关系 | Agent 是手中紧握的工具，人全程参与每一步 | 人从"操作者"变成"Loop 设计者" |
| 工作焦点 | 逐条编写提示词 | 设计自动化工作流 |

### 3. 六大构成板块（5 块积木 + 1 层记忆）

由 Google Cloud AI 总监 Addy Osmani 系统阐述：

1. **自动化（心跳/定时任务）**：固定时间点或特定事件触发 Loop。手段：Claude Code 的 `/loop` 定时任务、cron 调度、hook 在 Agent 生命周期关键节点触发、推到 GitHub Actions 让电脑合上也能跑。重要模式 **`/goal`**：不按时间触发，而是持续运行到最初制定的标准完成为止，每轮由独立 Agent 判断是否完成。Claude Code 和 Codex 都支持。

2. **工作树（Worktree）**：基于 Git worktree，在同一仓库历史上切出独立工作目录，解决多个 Agent 同时写同一文件的撞车问题，每个 Agent 拥有独立工位互不干扰。

3. **Skill（技能）**：把开发者对项目的要求（代码风格、规范、架构、踩过的坑）打包展示给 Agent。Agent 每轮读取一次，无需重新输入。没有 skill 时 Loop 每轮从零推导；有了 skill，每轮吸取上次经验。

4. **插件和连接器**：决定 Loop 能否触碰真实工具。基于 MCP 协议，让 Agent 读 Issue Tracker、查数据库等。"只能看文件系统的 Loop 是小 Loop，能操作工具干活才是真正的 Loop。"

5. **子 Agent（Subagent）**：把"写代码"和"检查代码"职责拆分为两个独立 Agent。写代码的 Agent 给自己打分容易放水，需第二个独立 Agent（可配不同指令甚至不同模型）专门审查。子 Agent 是开发者敢放开监管的最大底气，也是 `/goal` 命令中判断目标是否完成的执行者。

6. **记忆层（Memory）**：整条流水线的脊椎。可以是 Markdown 文件、Linear 看板等活在单个会话之外的记录。**关键要求：记忆文件必须存在磁盘上，不能在上下文窗口里——"Agent 会忘，文件不会"。**

### 4. 典型应用流程

- **每天早上**：定时任务在代码仓库上自动启动
- **分诊阶段**：调用分诊 skill，翻查昨天 CI 失败记录、Open Issue、最近提交，写入 Markdown 或 Linear 看板
- **修复阶段**：对每个值得修改的问题，自动开独立 worktree，派子 Agent 起草修复方案
- **审查阶段**：再派第二个子 Agent 持项目 skill 文档和现有测试审查方案
- **提交阶段**：审查通过后连接器自动开 PR、更新工单；搞不定的放入 Triage 收件箱等人处理
- **状态管理**：state file 记录什么试过、什么通过了、什么还开着，第二天从停下的地方继续

→ 整个流程中，发现、分配、修复、审查、开 PR，开发者一个字都不用敲。

### 5. 三大风险（坑）

1. **验证仍然是你的活**："完成了"只是声明不是证明。拆分验证子 Agent 能让完成状态更可靠，但开发者发布确认能运行代码的核心职责不变。

2. **理解债务会滚雪球**：Loop 产出越快，实际代码与人能真正理解的代码差距越大。代码不是自己写的，这叫"理解债务"。丝滑的 Loop 只会让债务膨胀更快，除非开发者认真阅读并思考 Loop 产出。

3. **认知投降（最舒服的姿势最危险）**：Loop 自行运转时，意志不坚定的开发者易被诱惑不再自行判断、直接收下所有结果，作者称之"认知投降"。Loop 是解药还是毒药取决于人用判断力设计它来提效，还是用它逃避思考——"同一种行为，往往会得到相反的结果"。

### 6. 核心观点

- Loop 改变的是工作量，不让人从流程中消失（像公司能自行运转，老板也不能当甩手掌柜）。
- 底层逻辑通用：这些积木在 Codex 和 Claude Code 中都已成熟，设计好 Loop 后换哪个工具都能跑。
- 工作难点发生转移而非减少：Boris Cherny "我的工作是写 Loop"。
- 平衡是关键：搭好 Loop，但直接给 Agent 写提示词仍然有效，关键是找到正确平衡。
- Loop 越强，人越清醒：两个人搭完全一样的 Loop 可能得到完全相反结果——一人用它加速自己透彻理解的工作，另一人用它避开理解工作本身。
